# Retto Roulette 상용화 전 UI/UX·연출 감사 보고서

- 작성일: 2026-07-19
- 점검 대상: 현재 배포본 0.7.2, 첨부 이미지 3장, 현재 React/CSS 상태 모델
- 관점: 룰렛 연출자, 프로그램 UI 사용자, 스트리머
- 이번 문서의 범위: 문제 진단과 개편안. 0.8.0에서 P0/P1 핵심 항목을 구현하고 1080p·720p 방송 화면으로 재검증했다.

## 1. 결론

현재 화면은 기능 수는 충분하지만, 사용자가 해야 할 일을 기준으로 구조화되지 않았다. `전역 4단계` 안에 `명단 3단계`와 `설정 6단계`가 다시 들어가고, 정상 상태·선택 설정·고급 설정·진행 불가 상태가 모두 같은 분홍 카드 문법으로 표시된다. 그 결과 사용자는 추첨을 준비하는 것이 아니라 관리 양식을 끝까지 채우는 느낌을 받는다.

가장 큰 결함은 다음 네 가지다.

1. **4단계 경로의 현재/완료 단계명이 실제로 안 보인다.** 해당 글자색 `#4e342e`가 헤더 배경 `#4e342e`와 같아 대비가 1:1이다. 별도의 `header-pill` 상태명은 보이지만, 4단계 경로는 번호와 화살표만 남는다.
2. **후보가 0명인 막힌 상태를 정상 설정 화면처럼 보여 준다.** 동시에 `최대 1명`, `1명 선택`까지 남아 논리적으로 모순된다.
3. **완료 화면의 원판이 결과 패널에 덮이는 것처럼 잘린다.** 실제 원인은 결과 패널의 z-index가 아니라, 왼쪽 열보다 큰 원판을 `overflow:hidden`으로 자르는 레이아웃 계약 오류다.
4. **방송 진행자가 다음 행동을 통제할 수 없다.** 제외자 복구, 다음 회차, 새 명단, 새 세션, 직전 결과 취소가 서로 다른 곳에 숨거나 아예 없다.

따라서 `색을 더 넣고 아이콘을 붙이는 보수`만으로는 해결되지 않는다. 제품을 **세션 작업 공간 → 방송 무대 → 결과 및 다음 행동**의 세 층으로 재구성해야 한다.

## 2. 확인한 객관적 증거

| 증거 | 확인 결과 | 의미 |
|---|---|---|
| 첨부 이미지 1 | 4단계 경로의 현재/완료 단계명이 갈색 헤더에 묻힌다. 별도 `추첨 설정` 상태명은 보이지만 경로는 번호와 화살표만 남는다. | 단계 경로가 위치 안내가 아니라 장식 노이즈로 작동한다. |
| 배포본 1280×720 | 단계명 11px, 번호 원 17×17px. 첫 명단 카드가 y=155에서 높이 623px로 끝나 핵심 버튼이 첫 화면 아래로 잘린다. | 기본 방송 노트북 화면에서도 첫 행동을 한눈에 볼 수 없다. |
| 배포본 설정 화면 1280×720 | 문서 높이 2,072px, 뷰포트 높이 720px로 약 2.9화면이다. 버튼 19개와 입력 3개가 한 설정 문서에 동시에 존재한다. | “한 화면에서 정한다”는 설명과 실제 인지 부담이 다르다. |
| 첨부 이미지 2 | 전체 3명·현재 후보 0명·이전 당첨 제외 3명인데도 1명 추첨 설정과 나머지 선택지가 유지된다. | 복구가 먼저인 상태를 설정 상태로 잘못 표현한다. |
| 첨부 이미지 3과 1175×956 실측 | 무대 우측 516.8px, 원판 외곽 우측 663.5px로 약 146.7px가 잘린다. | 결과 패널이 겹친 것이 아니라 원판 크기와 열 크기 계약이 깨졌다. |
| 1280×720 완료 화면 실측 | 원판 약 298px, 결과 보드 약 448×425px, 제목 카드와 원판 약 109×85px 겹침. | 추첨 증거인 원판보다 기록 UI가 더 강하다. |
| 1920×1080 완료 화면 실측 | 문서 높이 약 1,280px, 다음 행동 영역은 y≈1,086부터 시작하고 원판 우측도 일부 잘린다. | 표준 1080p 송출에서도 스크롤 없이 운영할 수 없다. |
| `src/App.css:76-90`과 `src/rettoStock.css:80-86` | 활성 단계 글자와 헤더 배경이 모두 `#4e342e`. | 첨부 이미지 1의 가시성 문제는 취향이 아니라 CSS 결함이다. |
| `src/App.tsx:393-400` | 최대 당첨 수를 항상 최소 1로 강제한다. | 후보 0명인데 최대 1명이라는 모순의 직접 원인이다. |
| `src/App.tsx:299-312` | localStorage에는 당첨 기록만 저장한다. | 새로고침·브라우저 종료 시 명단, 상품, 설정, 현재 회차가 사라진다. |

참고 기준으로 WCAG 2.2는 일반 텍스트에 4.5:1 대비를 제시한다. 현재 4단계 경로의 활성 글자는 1:1이므로 이 기준에 미달한다. 17px 번호 원은 조작 버튼이 아니므로 포인터 대상 크기 기준을 직접 적용하지 않고, 여기서는 방송 화면의 인지 가독성 문제로 판단한다. ([WCAG 2.2](https://www.w3.org/TR/WCAG22/))

## 4. 추가 운영 결함: 리셋 체계만 고쳐서는 끝나지 않는 항목

52개 핵심 목록과 별도로, 실제 방송 운영에서 반드시 같이 해결해야 할 열세 항목이 있다.

1. **자동 다인 추첨에 중단·건너뛰기가 없다.** 첫 클릭에 전체 결과 계획을 고정하고 자동 공개하며(`App.tsx:663-700`), 진행 중 도구가 잠긴다. 잘못된 명단이나 방송 사고를 발견해도 멈출 수 없다. `다음 공개 일시정지`, `이번 회차 조기 종료`, `연출만 건너뛰기`가 필요하다.
2. **다인 당첨자의 상품 배정 흐름이 끊긴다.** 첫 당첨자에게 상품 뽑기를 시작하면 현재 사람 회차가 상품 회차로 교체되어 나머지 당첨자 버튼을 잃는다(`App.tsx:870-876`, `1382-1389`). 사람 회차 결과를 별도 세션 데이터로 유지하고 `상품 배정 대기 N명` 큐를 만들어야 한다.
3. **개별 제외 복구가 18명 제한과 충돌한다.** 목록의 `+N명은 명단 조정에서 확인` 안내는 사실과 다르다. 명단 조정에는 제외 상태가 없다. 검색 가능한 제외자 관리가 필요하다.
4. **직전 공개 결과 취소가 없다.** 방송 중 잘못된 대상이 포함됐거나 본인이 포기했을 때 `당첨 취소→후보 복귀/제외 유지→기록에 취소 사유 남김`이 필요하다. 기록을 조용히 삭제하면 공정성 논란이 생기므로 수정 이력을 남겨야 한다.
5. **가져오기 검수 수치가 부족하다.** 명단 확인은 처음 80명만 보여 주고 대댓글 제외 수는 보이지만 중복 제거 수는 조용히 사라진다(`ParticipantSetup.tsx:33-42`, `96-105`, `225-238`). `발견/대댓글 제외/중복 제외/최종 후보` 네 수치를 함께 보여 줘야 진행자가 시청자에게 차이를 설명할 수 있다.
6. **이름을 바꿔도 기존 제외 ID가 남을 수 있다.** 명단 저장은 기존 ID와 제외 상태를 보존하지만 편집 화면에는 제외 여부가 없다(`App.tsx:779-788`). 당첨자 이름을 다른 사람으로 수정하면 새 이름이 이유 없이 제외되는 것처럼 보일 수 있다. 저장할 때 `기존 제외 유지/모두 복귀`를 명시해야 한다.
7. **`검증 fnv1a`라는 표현은 검증 가능성을 과장한다.** 현재 값은 후보 목록 지문이지 난수와 결과를 독립 재현하는 증명이 아니다(`App.tsx:144-156`, `1427-1452`). 우선 `후보 지문`으로 이름을 낮추고, 실제 검증을 약속하려면 후보 스냅샷과 commit/reveal 설계를 별도로 도입한다.
8. **상품 재고를 개별 SVG 조각으로 펼친다.** 상품 종류와 수량이 크면 수천 개 옵션·조각이 생길 수 있다(`PrizeEditor.tsx:103-116`, `App.tsx:376-386`). 화면은 상품 종류 단위 조각을 쓰고 재고 수량은 확률·재고 모델로 계산해야 한다.
9. **기록 100건 제한이 한 회차 결과도 자를 수 있다.** 상품 다인 추첨 수에는 방송용 상한이 없지만 기록은 매번 최근 100건으로 자른다(`App.tsx:311`, `601`). 세션 결과는 전부 보존하고, 시네마틱 공개와 빠른 일괄 추첨의 상한을 따로 둔다.
10. **결과 데이터가 정상/무효/대체 상태를 표현하지 못한다.** 현재 상태 머신은 어떤 화면을 보여 줄지는 정의하지만, 실제 경품 운영에서 필요한 `valid/void/replaced`와 상품 배정 상태가 없다. 화면을 더 붙이기 전에 `세션→회차→결과 상태→상품 배정` 데이터 관계를 먼저 분리해야 한다.
11. **일부 후보 모드에서 개별 복귀가 실제 복귀로 이어지지 않을 수 있다.** 제외자를 복귀시켜도 이미 `poolIds`가 제한 수만큼 차 있으면 복귀한 ID가 현재 후보 풀에 추가되지 않는다(`App.tsx:340-367`, `814-819`). 성공 토스트와 실제 후보가 어긋나므로 복귀 시 풀 포함 여부를 직접 선택하거나 명시적으로 재구성해야 한다.
12. **제외가 늘어난 뒤 사전 후보 수가 현재 최대값보다 커질 수 있다.** `poolLimit` 상태는 기존 값으로 남아 입력의 `value > max`가 될 수 있다. 후보 수가 변할 때 값을 보정하고 변경 이유를 알려야 한다.
13. **후보 관련 변경이 조용한 재표본을 일으킨다.** 일부 후보 수 변경, 제외자 전체 복귀 등은 `poolIds`를 비운 뒤 effect가 새 후보를 무작위로 뽑는다. 누가 바뀌었는지 알리지 않으면 공정성 설명이 어렵다. `후보 다시 추리기`를 명시적 행동으로 만들고 전후 수와 시각을 기록해야 한다.

## 4-1. 반드시 보존할 현재 강점

개편이 전면 재작성으로 흐르지 않도록 아래 기반은 유지한다.

1. 추첨 버튼을 눌렀을 때 결과를 먼저 고정하고 연출은 그 결과를 공개한다. 공정성 설명의 좋은 기반이다.
2. 다트 복권은 한 발씩 진행하고 회차 조기 종료가 가능해 자동 다인 추첨보다 진행자 통제력이 좋다.
3. `CurrentRoundWinners`가 이번 회차의 전체 당첨자를 누적한다. 사용자가 요구한 공개 목록의 데이터 기반은 이미 있다.
4. 설정 화면을 연다고 즉시 추첨하지 않고 별도의 준비 상태를 거친다. 오클릭 방지에 유리하다.
5. 원판 중앙 `RETTO ROULETTE`, RettoStock 팔레트, 🍸💝 공식 마크는 브랜드 자산으로 보존할 가치가 있다.

## 5. 권장 첫 화면: 명단 페이지가 아닌 세션 작업 공간

명단을 미리 별도 페이지에서 작성해야 할 필수 이유는 없다. 후보 검증과 중복 정리는 필요하지만, 그것은 독립 절차가 아니라 선택한 추첨 목적의 입력 모듈이어야 한다.

기존 구조에도 의도는 있었다. 규칙을 고르기 전에 후보를 확정하면 미리보기와 최대 당첨 수를 계산하기 쉽고, 방송 중 후보가 몰래 바뀌는 것을 막을 수 있다. 그러나 이 두 장점은 `별도 첫 페이지`가 아니라 `준비 완료를 누르는 순간 후보 스냅샷 고정`으로 그대로 얻을 수 있다. 따라서 명단 검증은 보존하되, 명단 선행 페이지는 제거하는 편이 맞다.

```text
┌ Retto Roulette ─ 오늘 방송 / 자동 저장됨 ─ [새 방송] ┐
│                                                        │
│ 오늘 무엇을 뽑나요?                                    │
│ [👥 명단에서 당첨자 뽑기]  [🎁 정해진 사람에게 상품 뽑기] │
│                                                        │
│ 추첨 대상                                               │
│ [카페 페이지 붙여넣기] [이름 직접 입력] [이전 명단 열기] │
│ 현재 후보 70명 · 제외 0명 · 중복 2건 정리됨             │
│                                           [명단 관리]   │
│                                                        │
│ 결과: 3명        연출: 회전 룰렛                         │
│ 규칙: 동일 확률 · 당첨 후 다음 회차에서 제외             │
│ [세부 규칙]                         [추첨 화면 열기]     │
└────────────────────────────────────────────────────────┘
```

핵심은 한 화면에 모든 것을 우겨 넣는 것이 아니다.

- **항상 보여야 하는 것:** 무엇을 뽑는지, 현재 후보 수, 당첨 수, 연출, 제외 규칙, 시작 행동.
- **문제가 있을 때만 강하게 보여야 하는 것:** 후보 없음, 중복, 가중치 0, 재고 없음, 저장 복구.
- **기본적으로 접어야 하는 것:** 일부 후보 추리기, 개별 확률, 방송 제목, 색/소리 세부 조정.
- **별도 패널로 열어야 하는 것:** 전체 명단 편집, 제외자 관리, 기록, 상품 재고.

## 6. 상태 모델 재정의

| 상태 | 메인 화면 | 진행자 행동 | 시청자에게 보이는 것 |
|---|---|---|---|
| `empty` | 목적 선택 + 문맥형 입력 | 사람/상품 선택 | 송출하지 않음 |
| `configuring` | 핵심 3블록 + 접힌 고급 설정 | 후보·수량·연출 확인 | 송출하지 않음 |
| `blocked` | 원인 1개와 복구 행동 | 복귀/상품 보충/가중치 수정 | 송출하지 않음 |
| `ready` | 설정은 접고 무대와 주 버튼만 | 클릭 시 결과 고정 | 고속 회전 원판 + 공정성 배지 |
| `locking` | 도구 잠금, 긴급 중단만 유지 | 결과 변경 불가 | 감속 또는 다트 발사 |
| `boundary` | 조작 없음 | 필요 시 연출 건너뛰기 | 경계 양쪽 후보 클로즈업 |
| `reveal` | 무대 기하 고정 | 조작 없음 | 정지 조각 + 큰 당첨 이름 |
| `settled` | 결과 레일에 새 행 안착 | 다음 결과 또는 회차 완료 | 전체 당첨 현황 |
| `completed` | post-draw bar | 다음 추첨/복귀/설정/상품 배정 | 최종 당첨자와 명확한 종료 |
| `recovery` | 복구 가능한 마지막 세션 | 이어하기/새 세션 | 송출하지 않음 |

`blocked`를 `configuring`의 숫자 변형으로 취급하면 이미지 2가 다시 생긴다. 막힌 상태는 설정 카드가 아니라 복구 화면이어야 한다.

## 7. 완료 후 행동과 리셋의 공용 언어

| 행동 | 유지 | 초기화/변경 | 주 위치 |
|---|---|---|---|
| 다음 추첨 | 명단·설정·기록·제외 상태 | 현재 결과 무대 | 완료 바 1순위 |
| 이번 당첨자 후보로 복귀 | 명단·설정·기록 | 선택한 제외 ID | 결과 행 또는 완료 바 |
| 당첨자 모두 복귀 | 명단·설정·기록 | 모든 제외 ID | 후보 관리 |
| 설정 바꾸기 | 명단·기록·제외 | 수량·모드·규칙 | 완료 바 2순위 |
| 새 명단으로 교체 | 설정·기록(선택) | 명단·제외·현재 회차 | 완료 바 또는 헤더 |
| 새 방송 세션 | 저장 여부를 먼저 선택 | 현재 회차·명단·제외·기록 | 헤더 |
| 모든 로컬 데이터 삭제 | 없음 | 세션·기록·설정 전체 | 설정의 위험 구역 |

버튼 라벨에 결과를 포함해야 한다. `초기화` 대신 `당첨자 3명 모두 후보로 복귀`, `현재 결과만 지우기`처럼 쓴다.

## 8. 필요한 시각 요소와 표지

장식용 캐릭터 이미지를 늘릴 필요는 없다. 아래처럼 판단을 돕는 기능성 시각 요소가 먼저다.

1. **목적 표지:** 사람 추첨은 `사람 토큰→원판`, 상품 추첨은 `선물 상자→수령인`의 간단한 선화.
2. **입력 표지:** 카페 문서에서 이름 칩이 추출되는 작은 도식. 무엇을 붙여넣는지 즉시 이해시킨다.
3. **연출 썸네일:** 회전 룰렛의 포인터, 정면 다트 탄착, 마블 결승선의 실제 무대 축소 이미지.
4. **후보 상태 바:** `저장 70 / 현재 후보 67 / 제외 3`을 산술 관계가 보이는 막대와 칩으로 표현.
5. **제외 상태:** 분홍 선택색과 다른 호박색 일시정지 아이콘. `삭제`가 아니라 `다음 추첨 잠시 제외`임을 보여 준다.
6. **차단 상태:** 경고 삼각형 + 원인 문장 + 하나의 복구 버튼. 다른 설정은 배경으로 낮춘다.
7. **공정성 배지:** `후보 67명 · 동일 확률 · 클릭 시 결과 고정`을 방송 중 항상 읽을 수 있는 크기로 표시.
8. **당첨 조각 표지:** 정지 조각의 색을 이름 리본과 결과 행까지 이어 인과관계를 유지.
9. **결과 레일:** 번호, 이름, 상품, 제외/복귀 상태만 담고 프레임 장식은 최소화.
10. **방송 안전영역:** 개발·QA용 720p/1080p safe-area 가이드. 제품에서는 숨기되 자동 시각 테스트에 사용.

RettoStock 팔레트와 🍸💝 공식 마크는 유지한다. 다만 분홍색 하나로 선택·오류·제외·당첨을 모두 표현하지 않는다. 핑크는 브랜드/선택, 레몬은 주목, 민트는 완료, 호박은 제외/주의, 갈색은 본문이라는 의미를 고정한다.

## 9. 연출 순서 개편

회전 룰렛과 다트 복권의 샷·물리·카메라·사운드·해상도 계약은 별도 문서 `DRAW_DIRECTION_SPEC_2026-07-19.md`에 상세히 정의한다. 아래는 그 핵심 순서다.

### 회전 룰렛

1. 추첨 화면 진입: 0속도에서 빠르게 가속.
2. 준비 상태: 이름을 읽을 수 없을 정도의 안정된 고속 회전.
3. 버튼 클릭: 결과와 후보 지문을 즉시 고정하고 공정성 배지에 표시.
4. 감속: 포인터 근처만 점진 확대.
5. 경계: 양쪽 후보 이름을 보여 주며 마지막 미세 이동.
6. 정지: 0.9~1.2초 원판과 당첨 조각을 그대로 유지.
7. 공개: 원판을 지우지 않고 이름 리본과 효과를 겹침.
8. 안착: 이름이 결과 레일의 새 행으로 이동.

### 다트 복권

1. 원판은 준비 상태부터 고속 회전.
2. 클릭과 동시에 결과를 고정하고 정면 다트 발사.
3. 충돌 전에는 감속하지 않음.
4. 탄착 시 다트가 원판과 같은 각속도로 함께 회전.
5. 경계에 꽂혔을 때만 0.35~0.5초 짧은 경계 확인.
6. 다트가 박힌 원판이 12시로 감속·정렬.
7. 정지 후 최종 당첨자 공개와 결과 레일 안착.

## 10. 외부 서비스에서 가져올 패턴

- [Wheel of Names FAQ](https://wheelofnames.com/faq/sound)는 결과 대화상자에 명시적인 `Remove` 행동을 두고, 항목과 설정을 자동 저장하며 전체 화면을 지원한다. Retto Roulette도 제외를 숨은 전역 옵션이 아니라 각 결과에 연결된 행동으로 보여 줘야 한다.
- [Picker Wheel](https://pickerwheel.com/)은 일반·제거·누적 모드를 구분하고, 개별 선택/해제, 전체 선택/해제, 결과, 전체화면, 저장/열기를 제공한다. `당첨 후 제외`를 규칙 문장만이 아니라 현재 작동 모드로 표현할 근거가 된다.
- [Orbit Picker](https://picker.cybereasy.net/)는 결과 직후 `Spin again / Remove winner / Copy result`를 한곳에 둔다. 완료 뒤 다음 행동을 결과 옆에 모아야 한다는 좋은 패턴이다.
- [RANDOM.ORG Drawings](https://www.random.org/guides/draws/)는 참가자 검증과 결과 기록을 핵심으로 둔다. Retto Roulette도 후보 수·규칙·클릭 시각·결과를 한 장의 감사 기록으로 남기되, 이를 무대보다 크게 보여 주지는 않아야 한다.
- Nielsen Norman Group의 [10 Usability Heuristics](https://media.nngroup.com/media/articles/attachments/Heuristic_Summary1-compressed.pdf)는 시스템 상태 가시성, 사용자 통제와 자유, 기억보다 인식을 강조한다. 현재 숨은 제외 복구와 모호한 초기화는 이 세 원칙을 동시에 위반한다.

외부 제품을 그대로 복제할 필요는 없다. 공통적으로 검증된 것은 `입력→실행→결과→제거/다시 실행`이 한 흐름에 있고, 복구 행동이 결과 가까이에 있다는 점이다.

## 12. 구현 우선순위

### P0 — 방송 사고와 논리 모순 제거

1. 완료 화면 원판 절단과 제목 겹침 제거.
2. 720p·1080p 라이브 화면을 스크롤 없는 안전영역으로 고정.
3. 후보 0명일 때 최대 0, 수량 `—`, 원인별 복구 카드 표시.
4. 설정 화면에서 제외자 개별/전체 복귀.
5. `누구 중에서`를 필수 단계에서 후보 상태 카드로 축소.
6. 완료 바에 다음 추첨·복귀·설정 변경·새 명단을 모음.
7. 정지 조각 0.9초 이상 유지, Hero가 원판을 지우지 않도록 수정.
8. 실제 result docking 구현, 두 번의 레이아웃 점프 제거.
9. 단계표의 1:1 대비와 17px 표지 수정.
10. 세션 자동 저장과 새로고침 복구.
11. 자동 다인 추첨 일시정지/조기 종료 안전 제어.
12. 사람 다인 당첨자의 상품 배정 큐 보존.

### P1 — 정보 구조 개편

1. 목적 선택과 문맥형 입력을 합친 세션 작업 공간.
2. 설정 6구역을 핵심 3블록+고급 접힘으로 압축.
3. 원본 명단 편집과 현재 제외자 관리 분리.
4. 상태별 시각 문법과 기능성 썸네일 도입.
5. 결과 레일과 post-draw 행동 재배치.
6. 공정성 배지와 회차 감사 기록.
7. 직전 결과 취소와 수정 이력.

### P2 — 상용 운영 완성도

1. 저장된 세션/명단 열기, 가져오기/내보내기.
2. OBS용 무대 전용 모드와 진행자용 별도 제어 화면.
3. 애니메이션·음향 속도 프리셋과 모션 감소 옵션.
4. 긴 명단 검색, 가상 목록, 키보드 조작, 포커스 복귀.
5. 720p·1080p·세로 화면 시각 회귀 테스트.

## 13. 승인 기준

다음 조건을 모두 만족해야 상용 프로그램 수준의 1차 개편으로 본다.

- 첫 사용자도 10초 안에 `사람 추첨`과 `상품 추첨` 중 시작점을 구분한다.
- 일반 사람 추첨은 목적 선택 후 명단 붙여넣기, 수량, 연출의 세 판단만으로 시작한다.
- 후보 0명 상태에서 잘못된 `1명` 수치와 무의미한 연출 설정이 보이지 않는다.
- 제외된 사람은 설정과 결과 양쪽에서 이유를 확인하고 복귀시킬 수 있다.
- 720p와 1080p에서 원판, 포인터, 현재 당첨자, 전체 당첨자, 다음 행동이 서로 가리지 않고 첫 화면에 들어온다.
- 정지 조각과 당첨 이름의 인과관계가 최소 0.9초 유지된다.
- 자동 다인 추첨을 진행자가 안전하게 멈출 수 있다.
- 새로고침 후 마지막 준비 상태를 복구할 수 있다.
- `다음 추첨`, `당첨자 복귀`, `설정 변경`, `새 방송`의 데이터 유지 범위를 버튼만 보고 예측할 수 있다.
- 핵심 텍스트 대비와 조작 대상 크기가 WCAG 2.2 기준을 충족한다.

## 14. 최종 판단

첨부 이미지 1은 실제 색상 충돌 때문에 읽히지 않고, 이미지 2는 정보가 많아서가 아니라 중요도·상태·예외가 같은 문법으로 펼쳐져 난잡하며, 이미지 3은 결과 패널이 겹친 것처럼 보이지만 실제로는 원판 크기와 그리드 열 크기의 계약이 깨져 잘린다.

개편의 중심은 다음 한 문장이다.

> 준비할 때는 필요한 결정만 보이고, 추첨할 때는 원판과 긴장감만 보이며, 끝난 뒤에는 당첨자와 다음 행동만 보이게 한다.
