# Retto Roulette Operations

## 1. 운영 구조와 지원 범위

Retto Roulette는 GitHub Pages에서 실행되는 완전한 정적 웹 앱이다. 서버, 데이터베이스, Cloudflare Worker, 네이버 로그인 연결, 외부 AI API를 사용하지 않는다. 약 10명의 스트리머가 동시에 접속해도 각 브라우저에서 독립 실행되므로 공유 서버의 경합이나 무료 한도 과금이 없다.

시청자는 별도 클라이언트로 회차를 공동 조작하지 않고 스트리머가 송출하는 동일 화면을 본다. 다음 기능은 현재 지원 범위 밖이다.

- 여러 진행자 또는 여러 탭 사이의 회차 동기화
- 다른 기기에서 진행 중 회차 이어받기
- 새로고침 뒤 방송 세션·현재 회차·애니메이션 중간 상태 자동 재개
- 서버 기반 감사 로그와 계정별 권한

같은 브라우저의 여러 탭은 단일 `localStorage` 기록을 공유하지만 회차 상태를 조정하지 않는다. 마지막으로 저장한 탭의 기록이 우선할 수 있으므로 한 방송은 한 탭에서만 진행한다.

## 2. 데이터 수명과 복구

| 데이터 | 저장 위치 | 수명 | 복구 정책 |
|---|---|---|---|
| 카페에서 붙여넣은 원문 | 브라우저 메모리 | 명단 서랍을 닫을 때까지 | 저장·전송하지 않음 |
| 참여자·상품·현재 설정 | 브라우저 메모리 | 현재 탭 | 새로고침 뒤 새 준비 상태로 시작 |
| 방송 세션·현재 회차 | 브라우저 메모리 | 현재 방송 화면/탭 | 같은 설정의 다음 회차에는 누적 보드를 유지하지만 새로고침 뒤 자동 재개하지 않음 |
| 확정 후 미공개 결과 | `localStorage: retto-roulette-pending-result` | 기록 저장 완료 또는 다음 시작 복구까지 | 애니메이션은 재개하지 않고 중복 ID를 제거해 `공개 전 확정 복구` 기록으로 병합 |
| 당첨 기록 | `localStorage: retto-roulette-history` | 최대 100건 | 유효한 레코드만 읽고 CSV로 내보낼 수 있음 |

결과는 버튼 클릭에서 먼저 확정되고 pending 저장을 시도한다. pending 저장이 거부되면 앱은 결과 확정을 취소하지 않고 탭을 닫지 말라는 경고를 표시한다. 공개된 기록 저장이 거부되어도 추첨은 계속하며 한 번만 경고하고 CSV 보관을 안내한다. 정상 공개에서는 당첨 기록 저장이 성공한 뒤에만 해당 pending 항목을 지운다.

## 3. 코드로 고정된 입력·저장 범위

| 항목 | 범위 |
|---|---|
| 보존 당첨 기록 | 최근 100건 |
| 사람 추첨의 상품명·상품 추첨의 받을 사람 | 최대 40자 |
| 방송 제목 | 최대 50자 |
| 상품 이름 | 최대 60자 |
| 상품 한 행의 재고 | 0~999개 |
| 참여자·상품 추첨권 | 0~99장 |

재고 0개와 추첨권 0장은 현재 추첨 후보에서 제외된다. 명단 총원에는 별도 하드 제한을 두지 않으므로 대규모 방송에서는 실제 대상 브라우저로 준비 화면과 방송 연출을 확인한다.

## 4. 배포 절차

1. 변경 범위와 하위 호환성 영향을 검토한다.
2. 버전을 올리고 `DEVELOPMENT_LOG.md`를 갱신한다.
3. 아래 검증을 실행한다.

```bash
npm test
npm run pages:build
npm run pages:verify
git diff --check
```

`pages:build`는 TypeScript/Vite 생산 빌드 뒤 `dist/`를 `docs/`에 동기화하고 오래된 해시 자산을 제거한다. `pages:verify`는 HTML의 `/rettoroulette/` 경로, JS/CSS 존재 여부, `dist`와 `docs`의 내용 일치를 확인한다.

4. 변경 보고와 테스트 결과를 사용자에게 제시한다.
5. 사용자 승인 뒤 커밋하고 `main`에 푸시한다.
6. GitHub Pages 반영 뒤 다음 명령으로 HTML과 해시 자산을 확인한다.

```bash
npm run pages:health
```

## 5. 배포 후 스모크 테스트

- 첫 화면에 준비 설정과 움직이는 샘플 원판이 함께 보인다.
- 사람 명단이 비어 있으면 왼쪽 `명단` 행에 `명단 추가`가 보이고, 우하단 시작 버튼은 중립 비활성 상태다.
- 카페 원문 붙여넣기와 직접 입력이 모두 명단 서랍에서 동작한다.
- 저장 전 명단 수정이 미리보기에 반영되지만 실제 기록·제외·재고는 바뀌지 않는다.
- 회전 룰렛과 다트 복권이 방송 진입 즉시 가속하고, 약 0.9초의 가속 중에는 주 버튼이 비활성화되며 고속 도달 뒤 클릭 순간 결과를 고정한다.
- 회전 룰렛은 고속 제동 뒤 실제 경계 앞에서 느린 포토피니시를 거쳐 12시에 정지한다.
- 다트는 발사 전 표적을 숨긴 채 상단의 매번 달라지는 지점에 정확히 착탄하고 원판과 함께 회전한 뒤, 같은 로컬 경계를 12시에 정렬한다. 실제 경계 근접 탄착에서는 표지와 두 후보를 정지까지 유지하며 착탄 전에는 감속하지 않는다.
- 경계 후보의 왼쪽은 당첨 칸, 오른쪽은 직전 이웃 칸으로 원판의 실제 이동 방향과 일치한다.
- 결과 Hero와 전체 당첨자 레일이 원판을 가리지 않는다. 같은 규칙으로 한 명씩 두 번 뽑으면 누적 보드에 두 결과가 공개 순서대로 남는다.
- 공개 연출 선택에는 회전 룰렛과 다트 복권만 있고 마블 레이스는 나타나지 않는다.
- 브라우저 콘솔에 오류가 없고 1366×768 및 1920×1080에서 기본 문서 스크롤이 없다.

## 6. 롤백

1. 장애가 발생한 커밋의 직전 정상 커밋을 확인한다.
2. 이력을 파괴하는 reset 대신 되돌림 커밋을 만든다.
3. 정상 소스에서 `npm test`와 `npm run pages:build`를 다시 실행해 `docs/`도 함께 되돌린다.
4. 변경 보고와 승인 뒤 푸시한다.
5. `npm run pages:health`와 스모크 테스트로 복구를 확인한다.

## 7. 하위 호환성 계약

- 공개 주소 `https://11qaws.github.io/rettoroulette/`와 Vite base `/rettoroulette/`를 유지한다.
- 기존 `retto-roulette-history` 저장 키와 과거 레코드의 선택 필드 호환을 유지한다.
- 과거 `mode: marble` 기록은 계속 읽고 CSV로 내보낸다. 마블 엔진·타입은 보존하지만 현재 공개 연출 선택에서는 제외한다.
- `retto-roulette-pending-result`는 버전 1 형식을 사용하며, 이미 기록된 ID는 시작 복구에서 중복 삽입하지 않는다.
- 종료된 `#import=` URL 자동 수집 링크는 데이터를 가져오지 않고 붙여넣기 방식 안내만 표시한다.
- 상태·기록 스키마를 호환 불가능하게 바꿀 때는 Minor 버전을 올리고 지원 범위를 개발 로그에 명시한다.

## 8. 장애 확인 순서

1. `npm run pages:health`로 배포 HTML과 자산 해시 불일치를 확인한다.
2. GitHub Pages 원본이 `main /docs`인지 확인한다.
3. 로컬에서 `npm run pages:verify`로 오래된 자산이 남았는지 확인한다.
4. 추첨 오류는 `raffleStatus`, `spinKey`, `presentationRun`, `currentRound`, `broadcastSession` 순서로 확인한다. 이름·카페 원문·브라우저 저장 내용은 로그나 이슈에 복사하지 않는다.
5. 새로고침 뒤 결과가 의심되면 live 회차를 재개하지 말고 당첨 기록의 `공개 전 확정 복구` 표지와 pending 키의 유효성만 확인한다. 저장 원문은 공유하지 않는다.
6. 결과 무결성이 의심되면 새 회차를 이어서 진행하지 말고 공개된 기록을 CSV로 보존한 뒤 정상 버전으로 롤백한다.
