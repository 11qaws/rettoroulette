# Development Log

## 0.10.8 — 다트 탄착색과 최종 당첨색 동기화

### 문제와 원인

- 다트 결과 수식은 클릭 시 계산한 탄착 로컬 좌표를 최종 12시까지 그대로 운반하고 있었지만, `dart-flight` 종료 처리에서 탄착 상태와 다음 결합 회전을 같은 React 배치로 시작해 정확한 접촉 프레임이 실제로 그려진다는 보장이 없었다.
- 화면 좌표에 고정된 탄착 표적·섬광은 약 220ms 남아 있는데 원판은 즉시 계속 회전했다. 이미 원판에 박혀 함께 도는 다트와 달리 이 고정 표식 아래로 다른 조각 색이 지나가면서, 순간적으로 탄착색과 최종 당첨색이 다른 것처럼 보였다.

### 해결

- 비행 transition 종료 시 `impactRotation`을 먼저 렌더하고 두 animation frame 경계 뒤에 `dart-attached-proof` 회전을 시작한다. 이 경계는 별도 삽입·튕김 동작을 추가하지 않고 정확한 접촉 프레임 하나만 보장한다.
- 원판의 결합 회전이 시작되면 screen-space impact anchor를 즉시 숨긴다. 이후 접점은 원판 자식인 embedded dart만 표시하므로 원판이 움직이는 동안 옛 탄착 지점 아래로 다른 색이 지나가는 착시가 없다.
- 각 SVG 조각에 안정적인 `data-slice-index`와 `data-slice-color`를 넣어 브라우저에서 탄착점·박힌 다트·최종 12시의 실제 렌더 색을 같은 기준으로 계측할 수 있게 했다.

### 검증과 호환성

- 다섯 탄착각, 세 반지름, 동일 확률·일반 가중치·극단 가중치, 다섯 시작 회전과 지연 catch-up을 조합해 `탄착 조각 = commit winner = 최종 12시 조각` 불변식을 테스트한다.
- 실제 브라우저에서 서로 다른 두 회차를 계측했다. embedded dart 아래 조각은 결합 회전 전 구간과 최종 정지에서 각각 winner index `4`, `1`을 유지했고, 최종 색도 각각 lavender, lemon으로 탄착색과 일치했다. 고정 impact anchor는 결합 회전 시작과 함께 opacity `0`이 됐다.
- 결과 선택, 확률, pending 저장, 후보 스냅샷과 최종 회전 수식은 변경하지 않아 기존 기록·설정과 하위 호환된다.

## 0.10.7 — 방송 제목·후보 레일과 누적 표기 단일화

### 문제와 원인

- 준비 화면에서 정한 방송 제목이 실제 방송에서는 단계별 상태 문구로 바뀌거나 최종 viewport CSS에 가려져 시청자가 어떤 추첨인지 계속 확인할 수 없었다.
- 실제 후보 명단은 진행 도구 서랍에서만 볼 수 있었다. 그 목록은 전체 저장 명단과 제외 복귀 버튼을 위한 관리 UI라서, 후보 제한·0 가중치·당첨 제외가 반영된 실제 원판 스냅샷을 증명하지 못했다.
- 완료 화면의 상단 상태, 결과판 눈썹과 우측 배지가 `누적 N명/개`를 반복해 결과판의 정보 우선순위가 흐려졌다.

### 해결

- 상단 상태 바의 중심 문구를 단계별 진행 수 대신 불변 `stageTitle`로 바꿨다. 준비 화면의 사용자 제목 또는 대상별 기본 제목이 ready부터 motion·proof hold·completed까지 유지된다.
- 읽기 전용 `BroadcastCandidateRoster`를 추가하고 실제 원판과 같은 `displayNames`를 전달했다. 데스크톱은 `후보 레일 | 원판 | 결과 레일`, 900px 이하는 주 행동을 보존한 `원판 → 행동 → 후보 레일 → 결과 레일` 순서를 사용한다.
- 후보 10개 이하는 하나의 정적 목록만 렌더링한다. 11개부터 의미 목록과 `aria-hidden` 복제 목록을 한 트랙으로 이어 약 3초/행의 CSS 선형 루프를 사용하며, 연출 중에는 이동을 멈춘다. `prefers-reduced-motion`에서는 복제 목록과 애니메이션을 숨기고 원본 목록을 수동 스크롤한다.
- 완료 상태 숫자를 상단에서 제거하고 결과판 눈썹을 `🎉 방송 결과`로 바꿨다. 화면에 보이는 누적 수는 가장 오른쪽 `CurrentRoundWinners` 배지 한 곳만 남기되, 비가시 `aria-live` 발표는 접근성을 위해 유지했다.
- 새 구조의 CSS는 동결된 `rettoStock.css`나 기존 연출 파일에 섞지 않고 `rettoRoulette.liveInfo.css`와 컴포넌트 CSS로 분리했다.

### 검증과 호환성

- 10개는 목록 1개·애니메이션 없음, 11개는 원본 1개와 비가시 복제 1개·33초 선형 무한 루프임을 컴포넌트 테스트로 고정했다. 상품 재고 단위의 같은 이름도 중복 제거하지 않는다.
- 실제 브라우저 1920×1080과 1280×720에서 문서 스크롤 없이 제목·후보 레일·원판·주 행동이 보였다. 1280×720 완료 화면에서 후보 레일·391px 원판·307px 결과판의 교차 면적은 모두 0이고 행동 하단은 684px였다.
- 390×844에서는 가로 넘침이 없고 원판과 주 행동이 첫 화면에 남은 뒤 후보 레일과 결과판이 이어졌다. 11명 첫 회차 뒤 당첨 제외가 적용된 다음 ready에서는 원판과 후보 레일이 함께 10명으로 바뀌고 루프가 자동 해제됐다.
- 완료 화면에서 실제로 보이는 `누적 N명/개` 요소는 `.current-round-winners__count` 정확히 한 개였다. 추첨 결과 확정, 회차 상태, 세션 누적 배열, 저장 키와 CSV 스키마는 변경하지 않아 이전 브라우저 데이터와 호환된다.

## 0.10.6 — 준비 정보 재정렬과 단일 탄착 결과 증명

### 준비 작업공간

- 미리보기 루트가 실제 방송의 `무대 + 결과판` 2열 클래스까지 상속해 비어 있는 결과 열만큼 원판이 왼쪽으로 밀리던 원인을 분리했다. 시네마틱 phase 클래스는 유지하되 미리보기는 하나의 중앙 grid만 소유하고, 원판 크기는 모바일에서도 실제 캔버스 폭과 stage padding 안에서 제한한다.
- 설정 순서를 `방송 제목 → 추첨 대상 → 연출 → 명단 → 세부 설정`으로 바꿨다. 방송 제목을 세부 설정에서 꺼내 입력 즉시 미리보기 좌상단에 표시하고, 제목 변경은 진행 중인 미리보기 물리 주기를 다시 시작하지 않는다.
- 명단 행은 데스크톱과 좁은 컨테이너 모두 `명단 · N명 · 편집`을 같은 기준선에 둔다. 제외 복귀와 빈 명단 추가 행동도 같은 행 안에서 유지한다.

### 탄착과 결과 이름표

- 다트의 `dart-impact-contact` 정지 프레임과 다음 `requestAnimationFrame` 재부착을 제거했다. 비행 transition 종료 프레임에서 화면 다트와 embedded dart를 같은 좌표·크기로 교대하고 원판 결합 회전을 바로 시작한다.
- 비행 다트와 embedded dart에 동일한 shaft 실루엣을 적용하고 원판 쪽 중복 contact 링을 제거했다. `dart-impacted` 카메라는 220ms 동안 유지한 뒤 `dart-attached` 풀백으로 이어져 한 번의 충돌이 두 번 박히는 것처럼 보이지 않는다.
- 실제 경계 결과의 승자 카드에는 이름 위 `WIN!`을 표시한다. 내부 정지에도 같은 스타일의 단일 당첨자 이름표와 `WIN!`을 proof hold부터 결과 화면까지 유지한다.

### 검증과 호환성

- Vitest 13개 파일 95개 테스트와 TypeScript/Vite 생산 빌드를 통과했다. 새 테스트는 설정 행 순서·방송 제목 단일 편집점, 두 다트의 공통 실루엣, 경계/내부 결과의 `WIN!` 문법을 고정한다.
- 실브라우저 1280×720과 1920×1080에서 문서 스크롤 없이 미리보기 원판과 캔버스 중심 X가 0.01px 이내로 일치했고, 제목 입력 중 미리보기 signature가 유지됐다. 390×844에서도 가로 넘침 없이 명단·인원·편집이 한 줄에 놓이고 원판 중심이 캔버스 중심과 일치했다.
- 다트 미리보기에서 `dart-launched → dart-impacted → dart-attached` 순서를 추적했다. 화면 다트와 embedded dart의 교대 프레임 중심 오차는 0.01px 미만이고, 충돌 카메라는 약 227ms 유지된 뒤 원판 결합 회전으로 이어졌다. 실제 내부 정지 화면에서도 `WIN! + 당첨자` 이름표와 누적 결과판이 함께 보임을 확인했다.
- `drawLabel`/`roundLabel`, 당첨 기록·pending·CSV 스키마, 난수 결정 시점, GitHub Pages 경로, 마블 기록 호환, 동결된 `src/rettoStock.css`는 변경하지 않는다.

## 0.10.5 — 실제 물리 연출 통합과 한 번에 한 명 추첨

### 다트·회전 룰렛의 공통 물리 계약

- 다트 복권의 발사 경로를 여러 타이머와 지터 구간에서 단일 `flight` 애니메이션으로 합쳤다. 화면 표적·정면 다트·embedded dart는 하나의 impact anchor를 사용하며, 클릭 뒤에는 경로를 다시 조정하지 않고 한 번에 꽂힌다.
- 발사 전 조준점은 난수 waypoint 사이를 smootherstep으로 이어 순간이동 없이 움직인다. 클릭 순간 마지막 렌더 좌표·원판 위상·각속도를 동결하고, 착탄 다음 프레임부터 다트가 원판 자식으로 함께 회전한다.
- 회전 룰렛의 근접 착지는 생성 단계부터 실제 경계 임계값 안에 두고 포토피니시를 최소 1.95초 확보했다. `A — 경계 — B`는 회전 룰렛에서 정지 약 2초 전, 다트에서 정지 1초 전 이름과 흐림이 함께 풀리는 시점부터 proof hold까지 유지한다.
- 라이브와 미리보기 모두 같은 `RouletteWheel`, 다트 aim/commit, reveal phase, 카메라 CSS를 사용한다. 미리보기 전용 당첨자 재조준과 원판 재마운트를 제거했다.

### 방송 중 필요한 만큼 한 명씩

- 준비 화면의 `당첨 인원/수량` 스테퍼와 대상별 winner-count 상태를 제거했다. 회전 룰렛과 다트 복권 모두 `클릭 1회 = 결과 1개`만 pending 저장·확정·공개한다.
- 자동 연속 공개 큐와 다트 전용 `awaiting-dart` 상태를 제거했다. 결과 뒤 `한 명 더 뽑기` 또는 `하나 더 뽑기`가 같은 방송 세션을 유지한 채 새 `ready` 상태를 만들고, 진행 여부는 스트리머가 매번 결정한다.
- 첫 추첨 전에는 `추첨 종료 · 새로 설계`, 결과 뒤에는 `이번 추첨 끝내기 · 설계로`를 유일한 보조 행동으로 고정했다. `end-broadcast` 상태 전이가 현재 원판과 방송 세션만 정리하고 명단·기록·당첨 제외·상품 재고는 보존한다.
- 진행 도구에는 확인이 필요한 `명단·상품·기록 모두 초기화`를 별도로 두었다. 방송 종료와 데이터 삭제를 같은 행동으로 오해하지 않도록 분리했다.

### 검증과 호환성

- Vitest 12개 파일 91개 테스트, TypeScript/Vite 빌드와 `git diff --check`를 통과했다.
- 1280×720 실브라우저에서 준비 화면의 당첨 수 컨트롤이 없고 문서 스크롤 없이 CTA가 보임을 확인했다. 회전 룰렛을 같은 세션에서 두 번 실행해 누적 보드가 1명에서 2명으로 늘고 두 번째 원판도 정상 정지했으며, 다트 복권 한 발도 같은 완료/계속/종료 도크로 끝났다. 준비·대기·결과의 액션 하단은 모두 720px 안에 남았다.
- 종료 뒤 명단 5명, 당첨 제외 2명과 전역 기록이 유지되고 새 방송 보드는 비워졌다. 전체 초기화 확인 뒤에는 명단·상품·기록과 설정이 모두 기본값으로 돌아갔다.
- 기존 `DrawRecord`/CSV/pending/history 저장 키, 과거 다인 결과의 `roundOrder`, 마블 기록 읽기, GitHub Pages `/rettoroulette/` 경로와 동결된 `src/rettoStock.css`는 유지한다.

## 0.10.4 — 다트 탄착 좌표를 결과의 단일 원본으로 통합

### 문제와 원인

- 이전 다트 연출은 당첨자를 먼저 고른 뒤 해당 조각을 탄착점으로 돌려 맞추는 경로를 포함했다. 화면의 조준점, 원판의 충돌 위상, 최종 12시 정지가 서로 다른 계산을 사용해 같은 결과처럼 보이도록 매번 보정해야 했다.
- 경계 착지는 한쪽 경계에 편향되어 있었고 내부 탄착이 부족했다. 회전 룰렛 역시 실제 임계거리보다 먼 위치까지 경계 연출로 취급했다.
- 전환 종료 이벤트가 누락되거나 `prefers-reduced-motion`이 활성화되면 다트 이름 공개가 빠지거나 착탄 뒤 상태가 다시 `approach`로 돌아갈 수 있었다.

### 물리 좌표와 연출 계약

- 추첨 대기 중 결과와 무관한 조준점을 계속 움직이고, 클릭 순간 마지막으로 실제 그린 탄착점·원판 각도·각속도를 한 프레임에서 동결한다.
- 고정 비행 시간 뒤 그 화면 좌표 아래에 도달하는 실제 조각을 당첨자로 확정한다. 화면 표적, 정면 비행 다트, 원판에 박힌 다트는 동일한 `--dart-impact-x/y`를 사용한다.
- embedded dart는 충돌 순간 원판 회전을 상쇄해 화면 탄착점과 같은 픽셀에 나타난 뒤 원판 자식으로 함께 회전한다. 최종 정지는 같은 원판 로컬 탄착점을 12시 축으로 운반할 뿐, 선택된 당첨자에게 원판을 다시 맞추지 않는다.
- 라이브 다트는 물리 커밋의 당첨 인덱스와 정규화 가중치 조각 서명이 현재 원판과 모두 일치할 때만 실행한다. 기존 리타기팅 경로는 명시적인 미리보기에서만 허용한다.

### 경계·포인터·예외 처리

- 회전과 다트 착지를 `near-start`, `near-end`, `interior`로 분리하고 실제 경계 거리 안에서만 경계 카드를 표시한다. 시작 경계는 오른쪽, 끝 경계는 왼쪽 당첨자로 표현한다.
- 다트 경계 카드는 착탄부터 정지까지 유지하며, 마지막 1초에 최종 12시 좌표로 이동해 이름을 공개하고 완전 정지 뒤에만 승자를 강조한다.
- 12시 포인터는 고정 핀과 침의 회전축을 동일 CSS 변수로 공유한다. 위쪽 핀은 움직이지 않고 아래 침만 흔들리며 최종 `0deg`에서 원판 중심축과 정렬된다.
- 전환 fallback에서도 다트 이름 공개 단계를 확정하고, 착탄 시 접근 타이머를 해제해 상태 역행을 막는다. 커밋 실패 뒤에는 새로운 조준 세션을 즉시 준비한다.

### 검증과 호환성

- 동일·가중 원판, 2~10명, 양쪽 경계와 내부 탄착, 클릭 시점 회전 변화에 대해 물리 탄착 조각과 최종 12시 당첨자가 일치하는 단위 테스트를 추가했다.
- 실제 브라우저에서 클릭 시 고정된 탄착 좌표가 비행 중 변하지 않고, 충돌 프레임의 screen target과 board anchor 중심 오차가 `0px`임을 확인했다. 최종 정지에서는 같은 x축으로 12시에 정렬됐다.
- Vitest 11개 파일 88개 테스트, TypeScript/Vite 빌드, `git diff --check`를 통과했다. 기존 저장소·CSV·pending 결과·GitHub Pages `/rettoroulette/` 경로와 RettoStock 원본 CSS는 변경하지 않는다.

## 0.10.3 — 12시 판정 침 고정과 전 화면 라이브 안전영역

### 문제와 원인

- 원판 좌표 계산은 잠긴 당첨 칸을 정확히 12시에 맞췄지만, 포토피니시의 포인터 애니메이션이 상단 핀을 축으로 끝 프레임을 `-9deg` 기울인 채 유지했다. 실제 침 끝이 계산 기준인 12시에서 옆으로 이동했고, 경계 안쪽 여유도 `0.25~0.8deg`에 불과해 최종 정지 화면에서 이웃 칸을 가리킬 수 있었다.
- 라이브 높이 제한이 첫 결과 전 `has-no-results-panel`과 결과 확정 `completed`에만 적용됐다. 첫 당첨 뒤 `다음 추첨`으로 돌아온 `ready + has-results-panel`은 기본 최소 높이에 끌려 1920×1000 뷰포트에서 액션 도크 하단이 1126px까지 내려가고, 셸의 `overflow: hidden`에 잘렸다.
- 공정성 문구와 정지 안내가 원판 위·아래의 별도 행을 차지하고, 원판 크기는 남은 세로가 아니라 `vh` 중심으로 계산됐다. 720p에서는 12시 침과 림이 무대 바깥으로 잘리고, 중간 폭에서는 원판의 내재 너비가 결과 레일까지 밀려날 수 있었다.

### 판정 증명과 반응형 무대

- 포인터 회전축을 선택 끝점인 `50% 100%`로 옮기고 최종 틱을 반드시 `rotate(0deg)`로 끝낸다. 끝점에는 작은 고정 판정점을 추가해 흔들림 중에도 실제 12시 좌표를 명확히 보인다.
- 자동 룰렛은 실제 당첨 경계를 천천히 통과하는 포토피니시를 유지하되, 마지막 증명 위치는 최소 4도 안쪽으로 이동한다. 폭이 8도보다 좁은 가중치 칸은 중앙에 멈춰 반대 경계를 넘지 않는다. 다트의 실제 경계 탄착 판정 `0.9deg` 이하는 그대로 유지한다.
- 라이브 전용 `rettoRoulette.viewport.css`를 새로 분리했다. 데스크톱 셸은 `헤더 / 상태 바 / 남은 캔버스` 그리드이고, 캔버스는 액션 행을 먼저 확보한 뒤 size container의 가로·세로 중 작은 값으로 원판을 정한다. 첫 추첨, 결과 확정, 누적 결과가 있는 다음 추첨 모두 같은 높이 계약을 사용한다.
- 원판 위·아래의 중복 문구는 라이브 무대에서 숨기고 상태 바와 액션 도크의 한 번만 남긴다. 원판 stage는 침 높이와 림 그림자만큼 대칭 안전 여백을 확보하며, 자동 경계 확대는 12시 상단을 원점으로 삼는다.
- 1180px 이하 데스크톱은 주 행동과 보조 행동을 두 행으로 바꾸고, 900px 이하에서는 결과 이력보다 액션을 먼저 배치해 하단에 고정한다. 모바일 결과 보드는 별도 내부 스크롤을 사용한다.

### 검증과 호환성

- 2~10명, 동일·가중치 원판, 모든 당첨 인덱스에서 최종 12시 아래 조각을 역산해 잠긴 `winnerIndex`와 일치하는 회귀 테스트를 추가했다. 좁은 가중치 칸의 중앙 정지도 함께 고정한다.
- Vitest 11개 파일 80개 테스트와 Pages 생산 빌드·산출물 검증, `git diff --check`를 통과했다.
- 실제 브라우저에서 1920×1000, 1600×900, 1366×768, 1280×720, 944×992, 768×1024, 600×900, 390×844를 검사했다. 첫 추첨 전과 누적 결과가 있는 다음 추첨 모두 주 액션이 뷰포트 안에 있고 침·림이 캔버스 안에 있으며, 원판과 결과 레일의 교차 면적이 0임을 확인했다. 두 번의 연속 추첨에서 최종 12시 역산 인덱스와 잠긴 당첨 인덱스가 일치했고, 고정 판정점의 가로 오차는 최대 0.016px이었다.
- RettoStock 원본 CSS, 결과 선확정·`spinKey + revealId` 콜백 차단, 다트 탄착 좌표, 방송 세션 누적 결과, 기록·pending·CSV 스키마, GitHub Pages `/rettoroulette/` 경로는 변경하지 않는다.

## 0.10.2 — 연속 추첨 안정화와 다트 경계 증명

### 원판 접촉과 연속 실행

- 600px SVG 안에서 조각 반지름이 258px에 머물러 핀과 당첨 영역 사이에 흰 띠가 보였다. 조각을 뷰박스보다 4px 오버스캔하고 핀 끝을 림 안쪽까지 겹쳐, 12시 핀이 색 조각을 직접 가리키도록 고쳤다.
- 연속 공개의 `transitionend`와 완료 콜백에는 어느 추첨 실행인지 판정할 ID가 충분하지 않았다. `spinKey + revealId` 실행 토큰을 원판 전환, 타이머, 중첩 RAF, 폴백 watchdog, App 완료 처리 전체에 적용해 늦게 도착한 이전 회차 이벤트와 중복 완료를 버린다.
- 새 회차는 직전 원판 각도를 DOM에 다시 확정하고 레이아웃 프레임을 만든 뒤 전환한다. 완료 뒤 내부 실행 상태도 `idle`로 되돌려 두 번째 자동 추첨과 자동 QA가 같은 상태를 읽도록 했다.

### 다트 복권의 탄착·결합·최종 확인

- 대기 원판과 비행은 1,080°/s까지 가속하고 이름을 흐리게 해 클릭으로 특정 후보를 유도하기 어렵게 했다. 결과가 먼저 고정된 뒤 생성하는 `DartShotPlan`은 상단 `-115~-65°`, 반지름 `0.58~0.80` 범위의 탄착점과 두 번의 작은 접근 지터, 회전량을 한 번만 만든다.
- 발사 전 고정 표적을 숨기고 접근 때만 탄착 표적을 보인다. 화면 다트는 지터를 0으로 수렴한 정확한 접점에서 즉시 사라지고 같은 좌표의 embedded dart가 한 박자로 나타난다. 72ms contact 뒤에는 다트가 원판 자식으로 결합돼 원판과 같은 물리 회전을 한다.
- 임의 위치 탄착 후 정수 회전으로 같은 화면 좌표에 돌아오던 계산을 수정했다. 꽂힌 다트의 원판 로컬 좌표는 그대로 보존하면서 원판이 감속해 최종 12시에 같은 경계와 당첨자를 정렬한다.
- 실제 경계 탄착에서 `경계선!` 표지와 양쪽 후보를 impact, coast, settled까지 유지한다. 최종 정지에서는 표지·후보를 12시로 이동하고 정확한 원판 좌표를 기준으로 확대하며, 당첨 조각과 왼쪽 당첨자 카드만 살짝 커지고 점멸한다. 다트 최종 증명은 1.5초 유지한다.
- 다트 대기 상태로 넘어갈 때 이전 embedded dart를 즉시 정리해 다음 발사 직전까지 이전 탄착물이 원판에 남지 않게 했다.

### 명단 입력과 검증

- 직접 입력에 `Shift+Enter로 한 번에 추가` 안내와 IME 조합 안전 단축키를 추가했다. 일반 Enter는 줄바꿈을 유지하며 예시 명단은 `아모레또, 유레카, 세나, 코코, 망징이`로 통일했다.
- Vitest 11개 파일 78개 테스트, TypeScript/Vite 생산 빌드와 `git diff --check`를 통과했다. 실제 1280×720 브라우저에서 Shift+Enter 5명 추가, 자동 룰렛 2명 연속 정지, 핀과 색 영역의 무간격 접촉, 다트의 서로 다른 탄착 좌표, impact → coast → settled 경계 표지 유지, embedded dart의 12시 최종 정렬과 당첨자 점멸을 확인했다.
- 공개 URL, Pages base, 기록·pending 저장 키, CSV와 과거 마블 기록 읽기 호환은 변경하지 않는다. 배포 전 Pages 산출물 생성·검증을 다시 실행한다.

## 0.10.0 — 포토피니시 추첨 무대와 방송 세션 결과 복구

### 문제와 원인

- 공개 준비 화면에 아직 마블 선택이 섞여 있어, 이번 릴리스에서 완성한 회전 룰렛·다트 복권의 차이와 주 행동이 흐려졌다. 명단이 없을 때 해결 행동도 왼쪽 설정 흐름과 떨어진 우하단에 있어 시작 차단 이유를 바로 고치기 어려웠다.
- 일반 룰렛은 하나의 감속 전환으로 곧바로 멈춰 경계 확대가 실제 물리와 분리돼 보였다. 반대로 다트는 고정 표적과 꽂힌 위치를 같은 좌표로 증명하지 못했고, 경계와 충분히 떨어진 착지에도 경계 연출이 나올 수 있었다.
- 결과 Hero와 보드가 원판 위에 겹칠 수 있었고, 결과 목록이 `CurrentRound`에만 의존해 한 명씩 여러 회차를 뽑으면 앞선 당첨자가 방송 화면에서 사라졌다.
- 결과는 클릭에서 확정되지만 공개·기록 저장 전에 새로고침하면 그 확정 사실을 설명할 영속 경계가 없었다. 공개 기록과 진행 중 애니메이션을 같은 복구 문제로 다루면서 라이브 재개 가능 범위도 모호했다.

### 준비 화면과 공개 연출 범위

- 공개 설정의 연출을 `회전 룰렛 | 다트 복권` 두 선택으로 제한했다. 마블 레이스는 공개 UI와 미리보기에서 제외하되 기존 엔진·타입, 과거 `mode: marble` 기록 읽기와 CSV 호환은 보존했다.
- 명단 0명에서는 왼쪽 `명단` 행에 채워진 `명단 추가`를 두고, 우하단 `방송 화면 열기`는 중립 비활성 상태로 남긴다. 저장 전 명단 초안의 변경·취소를 구분하고, 직접 입력의 줄바꿈과 중복 제외 피드백을 복구했다.
- 기본 설정은 대상·명단/상품·당첨 수·연출을 한 열로 맞췄다. 동일 확률/직접 지정과 당첨 후 제외/중복 허용은 접을 수 있는 `세부 설정`의 명시적 선택으로 바꿔 O/X 체크 해석을 없앴다.

### 회전 룰렛과 다트 물리

- 일반 룰렛은 후보가 둘 이상일 때 실제 경계 안쪽 `0.25~0.8°` 착지를 요청하고, 좁은 가중치 칸에서는 당첨 칸 폭의 90% 이내로 보정한다. 고속 대기 회전에서 1차 제동한 뒤 경계 `12~18°` 앞을 목표로 별도의 최소 `1.45초` 포토피니시로 이어지며, 두 단계의 경계 속도를 맞춰 재가속 없이 경계를 천천히 넘고 멈춘다.
- 경계 이벤트와 후보 이름은 계산된 착지 리드가 `0.9°` 이하이고 후보가 둘 이상인 실제 근접 장면에서만 발생한다. 화면 왼쪽에는 당첨 칸, 오른쪽에는 직전 이웃 칸을 배치해 원판의 실제 이동 방향과 일치시켰다.
- 다트는 탄착 전 고속 등속 회전을 유지하고 회전 룰렛의 정지 포토피니시를 사용하지 않는다. 화면 고정 12시 십자 표적, 정면 발사체의 종점, 원판 로컬 좌표의 embedded dart가 같은 CSS 좌표를 공유한다. 짧은 contact 프레임에서 일치점을 먼저 증명한 뒤 십자 표적은 즉시 사라지고, 다트와 접촉 링은 원판 자식으로 함께 회전한다.
- 방송 화면 진입 직후의 0.9초 가속 중에는 결과 확정/다트 발사 버튼을 `원판 가속 중…`으로 잠근다. 780°/s 순항 이벤트 뒤에만 활성화해 저속 클릭 때문에 제동 구간이 재가속되는 경로를 없앴다.
- 12시 당첨 포인터를 원판 상대 크기로 줄였다. 연출 중 제목·안내·결과를 숨기거나 별도 레일로 보내 원판이 추첨 무대의 주 시각 요소로 유지되도록 했다.

### 방송 세션 누적 결과와 pending 복구

- `BroadcastSession`을 추가해 `방송 화면 열기`에서 세션을 시작한다. 같은 규칙의 다음 회차는 세션 ID와 공개 순서의 결과 배열을 유지하고, 새 설정으로 방송 화면을 다시 열 때 새 세션을 만든다.
- `CurrentRound.results`는 이번 회차 진행과 남은 결과 수에만 사용하고, 시청자용 전체 당첨자 보드는 `BroadcastSession.results`를 사용한다. 한 명씩 반복 추첨해도 이전 당첨자가 남고, 당첨자별 상품 추첨 진입도 같은 누적 결과를 기준으로 한다.
- 결과 Hero를 원판 바깥의 전용 레일로 이동했다. Hero 공개 뒤 같은 자리의 누적 결과 보드로 전환해 1920×1080 방송 화면에서 원판을 가리지 않는다.
- 클릭에서 확정한 결과를 `retto-roulette-pending-result` 버전 1에 먼저 저장한다. 정상 공개는 `retto-roulette-history` 저장 성공 뒤 pending 항목을 소비하고, 시작 복구에서는 중복 ID를 제거해 최대 100건 기록에 합친 뒤 `공개 전 확정 복구`로 표시한다.
- 새로고침 뒤 방송 세션·현재 회차·애니메이션은 재개하지 않는다. 복구 대상은 확정 결과 기록뿐이며, 다중 탭 공동 진행도 계속 지원 범위 밖이다.

### 하위 호환성·운영과 검증

- 공개 URL, Vite `/rettoroulette/` base, 기존 `retto-roulette-history` 키와 CSV 열을 유지했다. `sessionId`는 선택 필드이므로 과거 기록을 그대로 읽는다. 공개 연출 선택과 결과 생애주기가 바뀌는 호환 가능한 구조 변경이라 버전을 `0.10.0`으로 올렸다.

- 실제 브라우저에서 1920×1080과 1366×768의 문서 스크롤 0, 원판과 Hero/결과 레일 교차 면적 0, 포인터 높이/원판 지름 9.65% 이하를 확인했다. 다트 contact 프레임의 화면 표적과 embedded dart 중심 오차는 0.001px 미만이었고, 가속 중 버튼 잠금과 순항 뒤 활성화도 확인했다.

## 0.8.0 — 경계 클로즈업과 방송용 결과 무대

### 결정적 순간을 보여 주는 추첨 연출

- 결과 선택과 공개를 `result-committed → motion-started → boundary-entered → boundary-crossed/impact-attached → rotation-stopped → proof-held` 상태로 분리했다. 버튼을 누른 순간 결과가 고정되고, 그 뒤의 카메라와 원판은 이미 고정된 결과를 증명하는 연출만 담당한다.
- 자동 룰렛은 고속 대기 회전에서 바로 감속 곡선으로 이어진다. 마지막에는 12시 포인터와 양옆 후보를 클로즈업하고 경계를 아슬아슬하게 통과한 뒤, 멈춘 장면을 0.95초 유지하고 이름을 공개한다.
- 다트 복권은 탄착 전까지 원판 속도를 유지한다. 정면 발사체가 화면 깊이축으로 접근해 회전 중인 판에 꽂히며, 화면 효과였던 다트를 실제 원판 회전 요소의 자식으로 교체해 탄착 뒤 원판·허브·다트가 한 물체처럼 함께 돈다.
- 다트의 경계 하이라이트는 탄착 순간에만 짧게 표시하고, 이후 원판이 12시에서 멈출 때까지 당첨자 이름을 숨긴다. 경계에 꽂혔다는 사실과 최종 당첨자 확인을 두 박자로 분리했다.

### 설정과 복구 흐름

- 설정의 `누구 중에서 뽑나요?` 블록을 `현재 추첨 대상` 요약으로 바꾸고, 1차 후보 무작위 추리기는 고급 설정 안으로 내렸다. 정상 사용자는 목적·수량·연출·규칙만 순서대로 확인하면 된다.

## 0.7.2 — 고속 대기 회전과 물리적으로 결합된 다트·허브

### 추첨 물리와 방송 연출

- 추첨 화면에 들어오면 자동 룰렛과 다트 복권 모두 정지 상태에서 0.9초 동안 가속한 뒤 이름을 읽기 어려운 약 780°/s로 계속 돈다. 버튼은 회전을 시작하는 버튼이 아니라 그 순간 결과를 고정하고 연출을 시작하는 버튼이다.
- 자동 룰렛은 클릭 직전의 정속 회전과 첫 감속 프레임의 속도를 맞췄다. 4회전 거리와 `p(t)=2t-t²` 감속 시간을 현재 속도에서 역산하므로 클릭 순간 재가속하지 않고, 마지막 경계 접근과 짧은 크리프를 거쳐 당첨 칸 안에 멈춘다.
- 다트는 대각선 화살 대신 화면 정면에서 출발해 원판이 가까워지는 1인칭 비행으로 바뀌었다. 탄착 전에는 판을 감속하지 않으며, 탄착 순간 화면 효과를 실제 `EmbeddedDart`로 교체한다.
- 원판, 중앙 `RETTO ROULETTE` 허브, 탄착한 다트를 같은 회전 요소 안에 배치했다. 허브는 처음부터 끝까지 판과 같은 각도로 돌고, 다트는 탄착 좌표를 판의 로컬 좌표로 한 번 변환한 뒤 판과 함께 회전·감속·정지한다.
- 경계선 하이라이트 여부를 조각의 절대 각도와 분리했다. 70명처럼 칸이 좁아도 결과와 무관한 24%의 사진 판정에서 짧은 `경계선!` 탄착 확인이 정상적으로 나온다.

### 설정 정보 구조와 모바일 흐름

- 설정을 `목적 → 후보/수령자 → 수량 → 연출 → 확률·중복 규칙 → 방송 제목`의 한 장 설계서로 재구성했다. 사람/상품의 인원수와 확률 모드를 각각 보존하고, 연출은 회전 룰렛·다트 복권·마블 레이스 세 카드로 같은 단계에 놓았다.
- 설정이 화면의 주 영역을 차지하고, 오른쪽 보조 미리보기에는 목적·범위·연출·규칙과 추첨 화면 진입 버튼만 남겼다. 전체 명단·실제 후보·이전 당첨 제외 인원도 서로 다른 수치로 표시한다.
- 모바일은 긴 설정을 끝까지 내리지 않아도 현재 범위와 연출을 확인하고 추첨 화면을 열 수 있는 고정 하단 도크를 제공한다. 헤더 단계는 번호만 남겨 브랜드가 줄바꿈되지 않으며, 상품 편집기의 불필요한 가로 스크롤도 제거했다.
- 모바일에서 버튼 클릭 순간 원판이 작아지던 우선순위를 바로잡았고, 화면낭독기용 룰렛 상태 문구를 `display:none`으로 제거하지 않도록 복구했다. 선택 버튼은 레또스톡 핑크 위에 짙은 글자를 사용한다.

### 검증

- 실제 브라우저에서 허브와 탄착 다트가 원판 DOM 안에 있는지, 탄착 뒤 다트의 화면 좌표가 판과 함께 이동하는지, 다트·자동 룰렛의 전체 상태 전이를 확인했다.
- 자동 룰렛은 대기 평균 787°/s, 클릭 직후 최대 778°/s에서 이후 계속 낮아지는 것을 프레임 표본으로 확인했다. 모바일에서는 추첨 전 273.9px 원판이 클릭 직후 축소되지 않고 278.4px 하이라이트로 이어졌다.
- 375×812와 1280×720 화면, 상품/가중치 편집, 새로고침 뒤 콘솔 오류 없음, Vitest 42개, TypeScript/Vite 생산 빌드, `git diff --check`를 통과했다.

## 0.7.1 — 방송 캔버스의 행동 계층과 결과 흐름 정리

### 전/후 시나리오

- 이전에는 추첨 대기·다트 대기·완료가 서로 다른 의미인데도 같은 버튼 격자를 일부 재사용했다. 720p 다트 대기에서는 빈 세 번째 열 때문에 `다음 다트`가 룰렛 중심에서 벗어났고, 완료 화면은 모바일에서 전체 결과보다 다음 회차 버튼이 먼저 나타났다.
- 이제 `BroadcastActionDock`이 `ready / awaiting / completed`를 각각 소유한다. 대기에는 추첨 시작 한 개, 다트에는 다음 발사와 현재 인원 마감, 완료에는 다음 추첨과 규칙·명단 변경만 나타난다. 결과 보드는 DOM과 모바일 시각 순서 모두 완료 행동보다 앞선다.
- 후보·상품이 없을 때 비활성 주 버튼으로 끝내지 않는다. 명단 준비, 당첨 제외 초기화, 가중치 조정, 상품 보충 중 현재 문제를 해결하는 동작으로 주 버튼이 바뀐다.

### 정보 구조와 방송 가독성

- 방송 헤더의 홈 링크를 정적 브랜드로 바꿔 실수로 명단과 회차를 새로고침하는 경로를 없앴다. 연출 중에는 밝은 비활성 버튼 대신 `연출 중 · 도구 잠김`만 표시하고, 다트의 다음 발사 대기에는 진행 도구를 읽기 전용으로 열 수 있다.
- 상태 바는 전체 키오스크 경로 대신 현재 상태·진행률·회차 규칙 한 줄만 보여 준다. 선택 입력인 방송 제목은 무대, 전체 결과 보드, WinnerHero 맥락, 기록과 CSV에 같은 회차명으로 남는다.
- 비어 있는 결과 보드는 0/N 안내 도크로 작게 유지한다. 결과가 생기면 확장하고 최신 행을 자동으로 보이며, 5명은 한 화면에, 6~10명은 2열에 배치한다. 긴 WinnerHero 이름도 길이에 따라 글자 크기를 줄여 생략 없이 표시한다.
- 판 중앙의 `RETTO ROULETTE` 허브와 기존 레또스톡 스킨은 변경하지 않았다.

### 막힌 기능 복구와 CSS 경계

- 기존 상품 상태·수정 함수는 있었지만 상품 추가 UI가 없어 `prizes=[]`에서 상품 추첨으로 갈 수 없었다. `PrizeEditor`를 추가해 상품 이름·재고·가중치·제외 이유를 설정 화면에서 바로 관리한다.
- 설정 그룹의 숨겨진 legend를 보이는 제목으로 바꾸고, 회차 제목·대상·인원·후보·연출 순서가 위에서 아래로 읽히도록 재배치했다. 설정 화면으로 돌아올 때 이전 스크롤 위치도 남지 않는다.
- 방송용 wheel 크기는 `App.css`만 소유한다. `RouletteWheel.css`의 부모 화면 결합 선택자를 제거해 12시 포인터 가림과 모바일 88vw 잘림의 원인을 없앴다. CSS import는 공통 기반 → 컴포넌트/레이아웃 → 룰렛 skin/shell 순서로 정리했다.

### 안정성·검증

- 표시용 후보 이름·가중치 배열을 메모해 룰렛 기하 계산이 불필요하게 다시 실행되지 않게 했고, 제외 ID 조회를 Set으로 바꿨다. 토스트 타이머 경합, 잘못된 localStorage 기록, 빈 상품명도 안전하게 처리한다.
- 첫 공개 배포의 1280×720 점검에서 버튼 도크가 두 줄을 차지해 룰렛 하단을 자르고, 긴 회차명을 넣은 빈 결과 보드가 내부 스크롤을 만드는 문제를 발견했다. 짧은 화면에서는 주·보조 행동을 한 줄로 압축하고 제목을 좌측 안전영역에 두었으며, 회차명과 `전체 당첨자` 역할을 분리해 두 문제를 제거했다.
- Vitest 40개와 TypeScript/Vite 생산 빌드를 통과했다. GitHub Pages 반영 뒤 1280×720·모바일 실제 화면과 전체 상태 전이를 다시 확인한다.

## 0.6.0 — 추첨 생애주기 상태와 회차 재시작 흐름

### 상태 모델과 불변식

- 화면 전환용 `roster / preflight / live` 추론을 `RaffleStatus` 전이로 교체했다. 상태는 `roster → configuring → ready → locking → presenting → completed`이며, 다트 복권 다인 추첨만 `awaiting-dart`를 거친다.
- 결과 선택은 `locking`에서 먼저 확정하고 `presenting`은 그 결과만 공개한다. 자동 룰렛/마블은 첫 클릭에서 회차 전체 계획을, 다트 복권은 발사 클릭마다 한 결과를 확정하는 기존 공정성 규칙을 상태 전이로 고정했다.
- 각 연출에는 `revealId`를 부여한다. 이전 애니메이션의 종료 콜백은 현재 연출 ID와 일치할 때만 결과를 기록하므로, 회차 교체·명단 초기화 뒤의 늦은 콜백이 새 회차를 오염시키지 않는다.
- 완료된 회차는 설정 화면과 섞어 그리지 않는다. 다음 회차 설정은 편집 가능한 현재 설정으로, 확정 결과는 완료 화면/기록으로 분리한다.

### 방송 화면과 복구 경로

- 모든 화면 상단에 `명단 → 설정 → 추첨 → 결과` 상태 경로를 표시하고, 방송 캔버스에는 현재 상태와 다음 가능한 행동만 둔다.
- 결과 확정 화면은 `같은 조건으로 다음 회차`, `설정 바꾸고 다음 회차`, `명단 바꾸기`를 항상 함께 보여 준다. 같은 조건은 제외/재고를 이어서 추첨 대기로, 설정 변경은 설정 화면으로, 명단 변경은 붙여넣기 화면으로 전이한다.
- 명단 편집에는 확인형 `명단 비우기`를 추가했다. 참여자·제외자·후보 풀만 초기화하고 기록과 상품 설정은 남긴다. 명단 교체를 닫으면 이전 완료 회차로 돌아간다.
- 다트 복권의 다음 다트 대기에는 `이번 회차 여기서 마치기`를 추가했다. 이미 공개한 결과는 되돌리지 않고 `endedEarly` 정보와 함께 완료 상태로 고정한다.
- 방송 중 설정을 드로어에서 바로 바꾸는 경로를 제거했다. 드로어는 명단·상품 현황·기록을 확인하는 보조 패널이고, 규칙 수정은 안전한 `configuring` 상태에서만 가능하다.

### 검증

- Vitest에 생애주기 전이 단위 테스트를 추가했다. 표준 흐름, 다트 대기/조기 종료, 활성 연출 중 명단·규칙 변경 차단을 확인한다.
- 실제 브라우저에서 직접 명단 → 설정 → 자동 룰렛 결과 확정 → 같은 조건 다음 회차 → 설정 재시도 → 명단 비우기, 그리고 다트 복권 2명 → 다음 다트 대기 → 조기 종료 → 명단 교체 취소 복귀를 확인했다.
- 1280×720에서 추첨 대기·결과 확정·다트 대기 각각의 주 동작이 첫 화면에 남는지, 375px 폭에서 가로 넘침이 없는지, 콘솔 오류가 없는지 확인했다.

## 0.5.0 — 통합 공정성 검증과 방송용 룰렛/다트 무대

### 이번 릴리스의 검토 기준

- **시작 흐름:** 명단을 받은 뒤에만 추첨 규칙을 정하며, 방송 시작 후에는 설정을 계속 노출하지 않는다. 결과 보드와 룰렛이 방송의 주인공이어야 한다.
- **결과 확정 시점:** 자동/마블은 첫 클릭에 회차 전체를, 다트 복권은 발사 버튼 클릭마다 정확히 한 결과를 확정한다. 애니메이션은 확정된 결과를 보여 주는 역할만 한다.
- **후보/재고 예외:** 사람의 중복 허용·제외, 당첨 인원 초과, 0 가중치, 가중치 비례 칸, 상품 재고 단위, 동일 상품 다수 재고, 다음 회차 설정 변경을 모두 분리해 처리한다.
- **검증 가능성:** 선택 시각과 공개 시각을 구분하고, 각 결과에 당시 후보 수·총 추첨권·후보 지문과 규칙을 남긴다. 화면 보드와 CSV가 같은 정보를 제공해야 하며, CSV는 스프레드시트 수식으로 해석될 수 있는 닉네임도 안전하게 내보낸다.
- **방송 연출과 상호작용:** 일반 룰렛은 무대를 크게 쓰고 마지막에 감속·확대한다. 다트 복권은 클릭 직후 발사되어 다트 시점으로 접근한 뒤 고정 탄착점에 착탄한다. 진행 도구는 무대를 밀지 않는 고정 오버레이여야 한다.

### 구현과 자체 확인

- 자동 룰렛/마블은 다인 결과를 클릭 시점에 한 번에 동결하고, 다트 복권은 매 발사 뒤 새 후보 풀로 다음 한 발을 동결한다. 당첨 뒤 제외가 켜진 다트 복권 다인 추첨에서 이전 당첨자가 다음 다트 후보에 남지 않는 것을 실제 흐름으로 확인했다.
- 사람의 중복 허용에서는 참가자 수보다 큰 인원을 선택할 수 있고, 상품은 재고 수량만큼의 독립 단위로만 당첨된다. 가중치 편집은 사람·상품 모두에서 지원하며 0은 명시적 제외다.
- 자동 룰렛의 마지막 하이라이트, 다트 복권의 발사·비행·접근·착탄, 결과 보드의 0/N→전체 당첨자 누적, 1280×720에서 시작 버튼이 보이는지 확인했다.
- 진행 도구를 열고 닫을 때 고정 레이어, 스크림, 본문 스크롤 잠금, ESC/포커스 복귀를 확인했다.
- 상품 3개 자동 추첨에서 선정 시각은 하나로 고정되고 공개 시각은 순차적으로 달라지며, 후보 수/총 추첨권이 3→2→1로 기록되고 재고가 0이 되는 것을 확인했다.
- 검증 완료: Vitest 28개 통과, TypeScript/Vite 생산 빌드 통과, `git diff --check` 통과, 직접 브라우저 E2E 흐름 확인.

### 배포 상태

- 사용자 요청에 따라 검증 완료 뒤 `main`과 `docs/` 배포본을 함께 갱신해 GitHub Pages에 반영한다.

## 0.4.0 — 클릭형 다트 피니시와 고정 결과 보드

- 룰렛의 당첨 결과는 `다트 발사` 또는 자동 회전 버튼을 누르는 순간 후보·규칙과 함께 확정한다. 다트 복권은 결과를 따로 고르는 기능이 아니라, 같은 결과를 빠르게 공개하는 룰렛 연출이다.
- 다트는 회전판의 고정된 12시 탄착점으로 향한다. 클릭 직후 다트가 출발하고 판은 착탄 전까지 선형 속도를 유지한다. 경계 착탄 뒤에만 감속해 당첨 칸 중앙에 멈춘다.
- 다인 추첨에서 자동 회전은 연속 공개를 유지한다. 다트 복권은 `다음 다트 발사`를 한 번씩 눌러 각 결과의 시작 시점을 방송에서 명확히 보인다.
- 방송 결과 보드는 추첨 전부터 `0/N`과 `추첨 대기` 행을 표시한다. 첫 결과가 나온 뒤 무대 크기나 열 구성이 바뀌지 않으며, 최신 당첨자만 명확히 강조한다.
- 현재 마블 레이스는 변경하지 않았다. 마블은 다음 단계에서 별도 Canvas 게임 스테이지로 교체한다.

### 확인·수정 기록

- 실제 브라우저에서 직접 입력 명단으로 다트 대기 → 클릭 직후 비행 → 확대·착탄 → 결과 보드 고정 흐름을 확인했다.
- 다인 다트에서 1/2 결과 뒤 `다음 다트 발사 (2/2)`가 나타나고, 두 결과가 중복 없이 보드에 누적되는 것을 확인했다.
- 자동 룰렛 2명 연속 추첨도 회귀 확인했다.
- 검증 완료: Vitest 15개 통과, TypeScript/Vite 생산 빌드 통과, `git diff --check` 통과.

### 배포 상태

- 사용자가 자동 배포 권한을 부여했다. 검증을 마친 변경은 `main`과 `docs/` 배포본을 함께 갱신해 GitHub Pages에 반영한다.

## 0.3.0 — 방송 집중 화면과 회차별 당첨자 보드

- 명단 준비 뒤의 화면을 `방송 준비`와 `방송 집중`으로 분리한다.
  1. 명단 준비: 카페 화면 붙여넣기 또는 직접 입력, 확인, 필요할 때만 다듬기
  2. 방송 준비: 명단 요약, 초기 추첨 규칙, 실제 룰렛/마블 미리보기
  3. 방송 집중: 설정을 접고 룰렛/마블과 회차 결과를 크게 표시
- 다인 추첨은 기존의 마지막 결과 하나(`latestResult`) 대신 `CurrentRound`에 결과 배열을 누적한다. 각 결과에는 선택적 `roundId`, `roundOrder`를 남겨 기존 localStorage 기록과 호환한다.
- 회차 결과 보드는 당첨자 전원을 번호·이름·상태로 생략 없이 보여 준다. 진행 중에는 방금 확정된 한 명만 강조하고, 완료 뒤에는 전원을 같은 당첨 상태로 표시한다.
- 다음 추첨 설정을 여는 동안에도 직전 회차 결과를 유지한다. 실제 다음 추첨을 시작할 때만 결과 보드를 새 회차로 교체한다.
- 명단·상품·CSV·기록·가중치·중복 당첨 옵션은 `진행 도구` 드로어로 이동한다. 방송 기본 화면에는 읽기 전용 규칙 요약만 남긴다.

### 확인·수정 기록

- 데스크톱 방송 화면에서 추첨 버튼이 첫 화면 아래로 밀리는 문제를 확인했다. 룰렛 크기와 무대 높이를 조정해 720px 높이에서도 시작 버튼을 함께 보이게 했다.
- 실제 브라우저 흐름으로 직접 입력 명단 6명 → 방송 준비 → 3명 룰렛 추첨을 확인했다. 결과 보드에 `Seioon`, `알레망`, `쌀과자또기`가 1·2·3번으로 함께 남고, 중복 당첨 방지로 다음 후보에서 빠지는 것을 확인했다.
- 진행 도구에서 다음 추첨 인원을 1명으로 바꿔도 3명 결과 보드가 유지되고, 새 추첨 시작 시 1명 결과 보드로 교체되는 것을 확인했다.
- 검증 완료: Vitest 12개 통과, TypeScript/Vite 생산 빌드 통과, `git diff --check` 통과, 방송 준비·집중·다인 결과·진행 도구 흐름을 브라우저에서 확인.

### 배포 상태

- 사용자 승인 후 `main`과 `docs/` 배포본을 함께 갱신해 GitHub Pages에 반영한다.

## 0.2.0 — 방송용 명단 준비 흐름 개편

- 기존 네이버 URL/Cloudflare 자동 수집은 회원 전용 게시글과 정적 호스팅 제약 때문에 제거한다.
- 참가자 준비는 브라우저 안에서만 처리하는 세 단계로 바꾼다.
  1. 카페 페이지 붙여넣기와 댓글 작성자 찾기
  2. 명단 확인
  3. 추가·삭제·순서 변경이 가능한 명단 다듬기
- 댓글 없이 진행하는 경우에는 첫 화면의 `이름 직접 입력하기`로 명단 다듬기 화면을 바로 연다.
- 기존 당첨 기록 localStorage 키(`retto-roulette-history`)는 유지한다.
- 0.2.0부터 네이버 URL 자동 수집과 `#import` 해시는 지원하지 않는다. 기존 링크는 앱을 열 수는 있지만 참가자를 자동으로 채우지 않는다.
- 시각 검토에서 명단 편집 하단의 시작 버튼이 기존 스크롤 위치를 유지하는 문제를 확인했다. 방송 화면으로 전환할 때 페이지 상단으로 이동하도록 보완한다.

### 확인·수정 기록

- Ctrl+A 복사 텍스트의 보조 형식에서 `첨부사진`·`스티커`가 이름으로 들어갈 수 있는 사례를 확인했다. 이미지 전용 댓글의 작성자는 유지하되, 보조 표기 자체는 닉네임으로 인정하지 않도록 회귀 테스트를 추가했다.
- 같은 닉네임이 답글과 원댓글 모두에 나타날 때 원댓글이 답글 표기보다 우선하도록 JSON 파서의 중복 처리를 통일했다.
- 명단 편집 하단에서 방송을 시작할 때 이전 스크롤 위치가 남는 사례를 시각 검토에서 확인했다. 방송 화면 전환 시 상단으로 이동하도록 수정했다.
- 사람 추첨 뒤 상품을 뽑는 흐름을 실제로 확인했다. 상품은 수량을 한 개 차감하고 기록에 수령자를 남기며, 룰렛·마블 공통으로 사람/상품 문구를 구분한다.
- 검증 완료: Vitest 12개 통과, TypeScript/Vite 생산 빌드 통과, 데스크톱·모바일 폭(375px 유효 폭)에서 가로 넘침 없음, 브라우저 콘솔 오류 없음.

### 배포 상태

- 코드 검토와 사용자 승인 전이므로 `docs/` 배포본과 원격 저장소에는 아직 반영하지 않았다.
